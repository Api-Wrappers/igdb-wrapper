API Wrapper Improvements Audit

TL;DR

The wrapper is already in a good baseline state: strict TypeScript is enabled, the public client covers the current endpoint registry, and bun run typecheck, bun test, and bun run build pass. Improve it incrementally, not with a rewrite. The highest-value next task is hardening runtime reliability and drift control: single-flight Twitch token refresh, endpoint metadata generation/validation, typed multi-query ergonomics, stronger error/debug behavior, and package smoke coverage.

1. Goal

Improve @api-wrappers/igdb-wrapper as a production API wrapper while preserving its main promise: typed APICalypse query construction, full raw IGDB escape hatches, and a small dependency footprint.

2. Codebase Understanding

  • Stack: TypeScript ESM/CJS library, Bun tests, tsdown build, strict tsconfig.json, Node >= 18, shared runtime via @api-wrappers/api-core.
  • Core runtime: src/client/IGDBClient.ts:23 owns public endpoint properties and high-level helpers. src/http/HttpClient.ts:29 wraps api-core with IGDB base URL, auth, retry, rate-limit, and error conversion. src/auth/AuthManager.ts:27 fetches and caches Twitch client-credentials tokens.
  • Query builder: src/query/QueryBuilder.ts:19, src/query/compiler.ts:3, src/query/fieldProxy.ts:18, and src/query/operators.ts:24 provide immutable fluent query construction with typed proxies.
  • Endpoint coverage: src/endpoints/registry.ts:1 maps camel-cased client keys to IGDB endpoint paths. src/client/IGDBClient.ts:23 manually mirrors that registry as class properties and constructor assignments.
  • Docs and roadmap: README.md, docs/, and ROADMAP.md already position the package well and track examples, typed multi-query, docs snippet tests, package smoke tests, and release readiness.
  • Test baseline: src/__tests__/igdb-client.test.ts:17 covers auth request shape, token caching after sequential calls, endpoint exposure, query compilation, raw multi-query, meta/protobuf, utilities, webhooks, and rate-limit error mapping.

Validation run during this audit: bun run typecheck passed, bun test passed with 10 tests, and bun run build passed.

Findings by Priority

High: Concurrent token refresh can stampede Twitch auth

Evidence: AuthManager.getAccessToken() returns the cached token when valid but calls #refresh() directly when the cache is absent or stale (src/auth/AuthManager.ts:46). There is no in-flight refresh promise. The test suite only proves sequential cache reuse (src/__tests__/igdb-client.test.ts:70).

Why it matters: A server process that starts several IGDB requests at once can make multiple identical Twitch token calls before the first one populates #cache. That burns rate budget and increases cold-start latency.

Recommended fix: Add single-flight refresh behavior inside AuthManager. Store an in-flight Promise<string> | null, reuse it while pending, clear it in finally, and preserve current error mapping.

High: Endpoint properties duplicate registry data manually

Evidence: IGDB_ENDPOINTS is the canonical map (src/endpoints/registry.ts:1), but IGDBClient manually declares and assigns every endpoint (src/client/IGDBClient.ts:23 through src/client/IGDBClient.ts:262). The test verifies parity at runtime (src/__tests__/igdb-client.test.ts:138) but does not remove the maintenance burden.

Why it matters: Adding or renaming endpoints requires changing registry, client property declarations, constructor assignments, docs, and tests. The current shape is usable but drift-prone as IGDB changes.

Recommended fix: Introduce a generated endpoint surface or a typed endpoint map helper. Keep existing named properties for backward compatibility, but move the source of truth into structured metadata that can generate or validate the public property declarations.

Medium: Raw multi-query is the biggest remaining string-heavy API

Evidence: client.multiQuery() accepts a raw body string (src/client/IGDBClient.ts:293). The README and roadmap explicitly say typed multi-query is not implemented yet.

Why it matters: Multi-query is valuable for real apps because it combines search/detail/count requests, but the wrapper currently loses its type-safety advantage at that boundary.

Recommended fix: Add a minimal typed multi-query builder that reuses QueryBuilder.raw() output rather than reimplementing APICalypse. Start with named result entries and count entries. Keep raw multiQuery(string) unchanged.

Medium: Query debug helpers always write to console

Evidence: QueryBuilder.debug() and QueryBuilder.explain() call console.log directly (src/query/QueryBuilder.ts:128 and src/query/QueryBuilder.ts:133), while the client runtime accepts an injected logger in config (src/client/config.ts:12).

Why it matters: Library code should avoid unconditional console output. It can leak query contents into logs and is harder for application owners to route or suppress.

Recommended fix: Prefer non-printing introspection methods as the stable API: raw() already exists; add inspect() or toJSON() for AST-style inspection. Mark console-printing helpers as debug-only in docs or route them through an optional logger if the builder can receive one without broad API churn.

Medium: Error mapping needs broader contract tests

Evidence: HttpClient.toIGDBError() maps api-core errors to IGDB-specific errors, including 401 and rate limits. The tests cover rate-limit mapping only (src/__tests__/igdb-client.test.ts:361) and docs describe auth/not-found/validation patterns.

Why it matters: Error shapes are part of the public API. Regressions here force users to parse messages or catch generic errors.

Recommended fix: Add focused tests for 401 to IGDBAuthError, non-JSON response bodies, plugin-error unwrapping, failed Twitch token fetch, validation errors for limit/offset/paginate/empty fields, and findById() not-found behavior.

Low: Release validation is close but not complete

Evidence: package.json has verify running bun run check && npm pack --dry-run. The roadmap tracks package-content smoke tests, and package.json exports point to dist/index.mjs, dist/index.cjs, and dist/index.d.mts.

Why it matters: A build can pass while the published tarball has broken CJS/ESM import behavior, missing docs, or unusable declarations.

Recommended fix: Add a package smoke test that packs locally, installs or imports the tarball in a temp fixture, checks both ESM and CJS entrypoints, and typechecks a small consumer snippet.

3. Recommendation

Do this now, but split it into two execution tasks. First harden reliability and release trust because those reduce user-visible risk without changing the public API. Second improve ergonomics with a typed multi-query builder and endpoint metadata generation. Do not rewrite the HTTP layer or replace @api-wrappers/api-core; the current runtime boundary is clean enough.

4. Scope

In Scope

  • Single-flight token refresh in AuthManager.
  • Endpoint metadata/source-of-truth cleanup that preserves current public client properties.
  • Minimal typed multi-query builder that composes existing QueryBuilder output.
  • Safer debug/introspection behavior for queries.
  • Expanded tests for auth, error mapping, validation, endpoint drift, multi-query, and package smoke behavior.
  • Docs updates for any public API additions or changed debug guidance.

Out of Scope

  • Removing raw APICalypse escape hatches.
  • Replacing @api-wrappers/api-core.
  • Changing package name, module format, or minimum Node version.
  • Browser-side OAuth flows; Twitch client secrets should stay server-side.
  • Large generated model redesign unless endpoint metadata work proves it is necessary.

6. Implementation Plan

  1. Add single-flight token refresh. In AuthManager, add a private in-flight refresh promise. Update getAccessToken() so concurrent callers share one refresh, then clear the promise in finally. Add a test that fires multiple concurrent IGDB requests against an empty cache and asserts exactly one Twitch token request.
  2. Expand error and validation tests. Add focused tests for 401 mapping, failed token fetch mapping, non-JSON/non-object response body formatting, validation failures, and findById() not-found behavior. Keep test fetch mocks local and deterministic.
  3. Create endpoint metadata as the single source of truth. Introduce metadata that binds the public key, endpoint path, model type name, and searchability. Use it to derive IGDB_ENDPOINTS and searchable paths. Preserve all existing exports and properties. If code generation is used, keep the generated output committed and add a check that fails on drift.
  4. Reduce manual endpoint-client drift. Either generate the declarations/assignments in IGDBClient from endpoint metadata or add a type-level/runtime drift test that ensures every metadata row has a client property and docs entry. Do not remove existing named properties.
  5. Add a typed multi-query builder. Build a small API around existing endpoint/query objects, for example named query entries and named count entries that compile to IGDB multi-query syntax. Reuse QueryBuilder.raw() bodies. Keep client.multiQuery(rawString) intact.
  6. Improve query introspection. Add a non-printing introspection method or document raw() as the preferred debug path. If keeping debug()/explain(), make their console behavior explicit and test their output lightly or route through configurable logging only if it can be done without contaminating query-builder construction.
  7. Add package smoke validation. Add a script or test that runs after build, packs the package, verifies expected files, imports ESM, requires CJS, and typechecks a minimal consumer snippet. Wire it into verify if it is fast and reliable.
  8. Update docs and roadmap. Update README/docs for single-flight behavior only if externally relevant, typed multi-query usage, debug guidance, and release validation. Mark completed roadmap items rather than duplicating them.

7. Code Quality Rules

  • Follow existing TypeScript style, strictness, and public export patterns.
  • Keep changes incremental and backward compatible.
  • Do not use any or weaken public types.
  • Reuse QueryBuilder, IGDBEndpoint, and existing query compiler primitives before creating new abstractions.
  • Avoid new runtime dependencies unless the package smoke workflow cannot be done cleanly with Bun/Node built-ins.
  • Do not remove raw request methods; they are part of the package's compatibility story.

9. Acceptance Criteria

  • Concurrent cold-start requests share one Twitch token refresh.
  • Existing public API remains source-compatible for current README examples.
  • Endpoint metadata has one clear source of truth and tests catch registry/client/searchability drift.
  • Typed multi-query supports at least named list queries and named count queries while raw multi-query remains available.
  • Error mapping and validation behavior are covered by focused tests.
  • Package smoke validation proves ESM, CJS, and declarations work from the built package.
  • bun run typecheck, bun test, bun run build, and package verification pass.

10. Risks and Notes

  • Public API compatibility: Endpoint metadata cleanup must not remove named properties or change existing endpoint paths.
  • Type complexity: Typed multi-query can become over-engineered quickly. Ship the smallest useful typed builder and keep raw escape hatches.
  • Generated code drift: If generation is introduced, add a check script. Generation without CI validation makes drift worse, not better.
  • Secrets: Do not add docs or tests that encourage exposing Twitch client secrets in browser code.
  • Build artifacts: Running bun run build rewrites dist/. Keep source changes and generated build output separated if committing.
